## 五、各迭代详细设计

### 5.1 Worker池健康检查增强

#### 问题分析

当前Worker崩溃后：

- 任务丢失，无法恢复
- 无自动重启机制
- 用户无感知，功能静默失效

#### 功能设计

**心跳检测机制**

```
配置项：
- heartbeatInterval: 2000ms（心跳间隔）
- heartbeatTimeout: 3000ms（超时判定，确保5秒内检测到崩溃）
- maxRetryCount: 2（最大重试次数，减少误判）

检测流程：
1. 每2秒向Worker发送ping消息
2. Worker响应pong
3. 超时3秒未响应标记异常
4. 连续2次异常触发重启（总计约7秒检测+恢复）
```

**崩溃恢复机制**

```
任务备份策略（任务入队时即保存）：
1. 任务入队时保存到backupQueue（Map结构）
2. 任务处理完成后从backupQueue删除
3. Worker崩溃时backupQueue保留所有待处理任务

恢复流程：
1. 检测到Worker崩溃
2. 从backupQueue获取待处理任务
3. 重启Worker实例
4. 按优先级重放任务
5. 记录崩溃日志

日志格式：
{
  timestamp: '2026-05-11T10:00:00Z',
  workerId: 'worker-1',
  error: 'TimeoutError',
  pendingTasks: 5,
  recoveryTime: 1200
}
```

#### 验收标准

### 5.3 内存泄漏检测修复

#### 问题分析

内存泄漏来源：

- 未清理的事件监听器
- 缓存无淘汰机制
- Worker内存不释放

#### 功能设计

**内存监控**

```
监控指标：
- usedJSHeapSize：已用堆大小
- totalJSHeapSize：总堆大小
- cacheSize：缓存占用
- listenerCount：监听器数量

采集频率：每30秒
阈值告警：heap > 100MB
告警处理：
1. 触发强制清理（优先清理大文件缓存 > 1MB）
2. 清理未使用的DOM引用
3. 记录清理日志
4. 若仍超阈值，通知用户"内存占用过高，建议刷新页面"
```

**定时清理机制**

```
清理策略：
1. 每5分钟执行一次清理
2. 清理项：
   - 过期缓存条目（超过TTL）
   - 无效监听器（DOM元素已移除的事件监听器）
   - 未使用DOM引用（WeakMap中无引用的对象）
3. 记录清理日志，包含清理项数量和释放内存大小
```

**监听器泄漏检测**

```
检测方法：
1. 注册监听器时记录 { target, type, handler, timestamp }
2. 定时检查 target 是否仍在DOM中（target.isConnected）
3. 若 target 已移除且超过30秒，标记为泄漏
4. 自动移除泄漏监听器并记录
```

| 文件                                            | 变更类型 | 说明                     |
| ----------------------------------------------- | -------- | ------------------------ |
| content/modules/plugins/CachePlugin.js          | 修改     | 内存监控集成             |

| content/modules/core/ResourceAcceleratorCore.js | 修改     | 定时清理、内存告警处理   |

### 5.4 位置感知加载完善

#### 问题分析

当前问题：

- 视口外资源过早加载
- 滚动时批量请求
- 无距离优先级

#### 功能设计

**IntersectionObserver优化**

```
配置项：
- rootMargin: '200px 0px'（预加载边界）
- threshold: [0, 0.1, 0.5, 1]（可见度阈值）

首屏优化：
- 首屏图片（前10张或视口内）立即加载，不延迟
- 首屏判断：
  - DOMContentLoaded 后 1 秒内的图片
  - 或 IntersectionObserver 首次触发时已可见的图片（兼容异步加载）

处理逻辑：
1. 观察所有图片元素
2. 进入边界时触发加载（首屏除外）
3. 可见度变化调整优先级
4. 完全可见后取消观察
```

**距离优先级队列**

```
优先级计算：
priority = basePriority + distanceBonus

distanceBonus:
- 视口内: +100
- 距离 < 500px: +50
- 距离 < 1000px: +20
- 距离 > 1000px: +0

队列处理：
1. 按优先级排序
2. 高优先级先处理
3. 动态调整队列
```

**滚动预测加载**

```
预测算法：
1. 检测滚动速度和方向
2. 计算未来1秒可能进入视口的元素
3. 提前加入加载队列
4. 滚动停止后取消未执行的预测任务

节流控制：
- 滚动速度 > 3000px/s 时暂停预测（快速滚动不预测）
- 最多预测 5 个元素
- 预测任务加入队列时优先级降级（低于用户主动触发）
```

### 5.5 LRU缓存淘汰算法

#### 问题分析

当前缓存：

- 无大小限制
- 无淘汰机制
- 内存无限增长

#### 功能设计

**LRU缓存实现**

```
配置项：
- maxSize: 50 * 1024 * 1024（50MB）
- maxEntries: 500（最大条目数）
- protectList: ['favicon.ico', 'logo.png']（保护列表）
- hotThreshold: 'dynamic'（动态热点判定，访问次数 > 平均访问次数 * 2）

数据结构：
Map<url, { data, size, lastAccess, accessCount, isHot }>

热点判定（每次淘汰前重新计算）：
1. 计算所有条目的平均访问次数 avgAccess
2. accessCount > avgAccess * 2 的条目标记为 isHot = true
3. 热点条目在淘汰时受保护

淘汰策略：
1. 新条目超限时触发淘汰
2. 优先淘汰非热点、非保护列表中的条目
3. 按 lastAccess 排序，删除最旧的
4. 删除直到容量 < maxSize * 0.9（预留10%缓冲）
5. 记录淘汰日志：淘汰条目数、释放内存大小
```

**缓存预热**

```
预热策略：
1. 页面加载完成分析热点资源
2. 后台静默预加载
3. 优先级低于用户操作
4. 预热失败不影响体验
```

**持久化缓存**

```
存储结构：
- localStorage: 配置和元数据
- IndexedDB: 大文件缓存

同步策略：
- 写入: 异步写入，不阻塞主线程
- 读取: 优先内存，其次IndexedDB
- 清理: 定期清理过期数据
```
